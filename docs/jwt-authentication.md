# JWT Authentication

## Overview

The FTGO platform uses JSON Web Tokens (JWT) for stateless authentication across independently deployed microservices. Each service validates JWT tokens independently using a shared signing key, eliminating the need for centralized session storage or inter-service authentication calls.

## Authentication Flow

### 1. Token Issuance

Tokens are generated by an authentication service (or API gateway) using `JwtTokenProvider`:

```java
@Autowired
private JwtTokenProvider jwtTokenProvider;

// Generate access token (1 hour TTL)
String accessToken = jwtTokenProvider.generateAccessToken(
    "user-123",           // user ID (sub claim)
    "john.doe",           // username
    List.of("ROLE_USER"), // roles
    Map.of("tenant_id", "t-456")  // custom claims (optional)
);

// Generate refresh token (24 hour TTL)
String refreshToken = jwtTokenProvider.generateRefreshToken("user-123");
```

### 2. Token Structure

**Access Token Claims:**
```json
{
  "sub": "user-123",
  "username": "john.doe",
  "roles": ["ROLE_USER", "ROLE_ADMIN"],
  "type": "access",
  "iss": "ftgo-platform",
  "iat": 1709568000,
  "exp": 1709571600,
  "tenant_id": "t-456"
}
```

**Refresh Token Claims:**
```json
{
  "sub": "user-123",
  "type": "refresh",
  "iss": "ftgo-platform",
  "iat": 1709568000,
  "exp": 1709654400
}
```

### 3. Request Authentication

Clients include the access token in the `Authorization` header:

```
GET /api/orders HTTP/1.1
Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...
```

### 4. Token Validation (per request)

The `JwtAuthenticationFilter` processes each request:

1. **Extract** — Reads `Authorization: Bearer <token>` header
2. **Validate** — Verifies signature (HS256), expiration, and issuer
3. **Type Check** — Ensures token is an access token (not refresh)
4. **Extract Claims** — Builds `FtgoUserDetails` from JWT claims
5. **Set Context** — Creates `JwtAuthenticationToken` in `SecurityContextHolder`

If any step fails, the filter passes the request through without authentication, and Spring Security's `FtgoAuthenticationEntryPoint` returns a 401 JSON response.

### 5. Token Refresh Flow

When an access token expires:

1. Client sends refresh token to the refresh endpoint
2. Server validates the refresh token
3. Server issues a new access token (and optionally a new refresh token)
4. Client uses the new access token for subsequent requests

```
Client                          Auth Service
  │                                  │
  │  POST /auth/refresh              │
  │  { "refreshToken": "..." }       │
  │─────────────────────────────────▶│
  │                                  │ Validate refresh token
  │                                  │ Generate new access token
  │  { "accessToken": "...",         │
  │    "refreshToken": "..." }       │
  │◀─────────────────────────────────│
  │                                  │
```

## Token Lifecycle

```
┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│   Created    │──▶│   Active    │──▶│   Expired   │
│  (iat set)   │   │ (valid JWT) │   │ (exp passed)│
└─────────────┘   └──────┬──────┘   └─────────────┘
                         │
                    ┌────▼────┐
                    │ Invalid │  (tampered, wrong key, etc.)
                    └─────────┘
```

| Token Type | Default TTL | Purpose |
|---|---|---|
| Access Token | 1 hour (3,600,000 ms) | API authentication |
| Refresh Token | 24 hours (86,400,000 ms) | Obtain new access tokens |

## Configuration

### Properties

All JWT properties are prefixed with `ftgo.security.jwt`:

| Property | Default | Description |
|---|---|---|
| `ftgo.security.jwt.secret` | *(none — required)* | Base64-encoded HMAC-SHA256 signing key. **Must be provided via environment variable.** |
| `ftgo.security.jwt.expiration-ms` | `3600000` (1 hour) | Access token expiration in milliseconds |
| `ftgo.security.jwt.refresh-expiration-ms` | `86400000` (24 hours) | Refresh token expiration in milliseconds |
| `ftgo.security.jwt.issuer` | `ftgo-platform` | Token issuer claim (`iss`) |
| `ftgo.security.jwt.enabled` | `true` | Enable/disable JWT authentication |

### Service Configuration

Each microservice configures JWT via `application.properties`:

```properties
# JWT secret — provided via environment variable (NEVER hardcode)
ftgo.security.jwt.secret=${FTGO_SECURITY_JWT_SECRET:}

# Token expiration (optional — defaults provided by library)
ftgo.security.jwt.expiration-ms=3600000
ftgo.security.jwt.refresh-expiration-ms=86400000
ftgo.security.jwt.issuer=ftgo-platform
```

### Signing Key Management

**Important:** The JWT signing key must NEVER be hardcoded in source or configuration files.

Recommended approaches:
1. **Environment variable:** `FTGO_SECURITY_JWT_SECRET=<base64-encoded-key>`
2. **Kubernetes Secret:** Mount as environment variable in pod spec
3. **Vault/AWS Secrets Manager:** Inject at runtime via sidecar or init container

Generate a secure 256-bit key:
```bash
openssl rand -base64 32
```

All services **must** use the same signing key to enable cross-service token validation.

## Accessing User Context

### Via SecurityUtils

```java
import com.ftgo.security.util.SecurityUtils;

// Get user details
Optional<FtgoUserDetails> user = SecurityUtils.getCurrentUserDetails();

// Get specific fields
Optional<String> userId = SecurityUtils.getCurrentUserId();
Optional<String> username = SecurityUtils.getCurrentUsername();
Collection<String> roles = SecurityUtils.getCurrentRoles();

// Get custom claims
Optional<Object> tenantId = SecurityUtils.getClaim("tenant_id");
Map<String, Object> allClaims = SecurityUtils.getAllClaims();

// Check authentication
boolean isAuth = SecurityUtils.isAuthenticated();
boolean isAdmin = SecurityUtils.hasAuthority("ROLE_ADMIN");
```

### Via Method Security

```java
import org.springframework.security.access.prepost.PreAuthorize;

@RestController
@RequestMapping("/api/orders")
public class OrderController {

    @PreAuthorize("hasRole('USER')")
    @GetMapping
    public List<Order> getOrders() { ... }

    @PreAuthorize("hasRole('ADMIN')")
    @DeleteMapping("/{id}")
    public void deleteOrder(@PathVariable String id) { ... }
}
```

### Via JwtAuthenticationToken

```java
import com.ftgo.security.jwt.JwtAuthenticationToken;
import com.ftgo.security.jwt.FtgoUserDetails;

@GetMapping("/api/profile")
public UserProfile getProfile(JwtAuthenticationToken auth) {
    FtgoUserDetails user = auth.getUserDetails();
    String userId = user.getUserId();
    String tenantId = (String) user.getClaim("tenant_id");
    // ...
}
```

## Error Responses

### 401 Unauthorized
Returned when no valid JWT is present:
```json
{
  "timestamp": "2026-03-04T20:00:00Z",
  "status": 401,
  "error": "Unauthorized",
  "message": "Authentication is required to access this resource",
  "path": "/api/orders"
}
```

### 403 Forbidden
Returned when JWT is valid but user lacks required role:
```json
{
  "timestamp": "2026-03-04T20:00:00Z",
  "status": 403,
  "error": "Forbidden",
  "message": "You do not have permission to access this resource",
  "path": "/api/admin/users"
}
```

## Testing

### Integration Tests

The security library includes comprehensive JWT integration tests:

- `JwtTokenProviderTest` — Token generation, validation, and claims extraction
- `JwtAuthenticationIntegrationTest` — End-to-end HTTP authentication flow
- `FtgoBaseSecurityConfigTest` — Security filter chain configuration

### Testing in Services

Use `@WithMockUser` for simple tests or generate real JWT tokens:

```java
@SpringBootTest
@AutoConfigureMockMvc
class OrderControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @Autowired
    private JwtTokenProvider jwtTokenProvider;

    @Test
    void authenticatedRequestSucceeds() throws Exception {
        String token = jwtTokenProvider.generateAccessToken(
            "user-1", "testuser", List.of("ROLE_USER"));

        mockMvc.perform(get("/api/orders")
                .header("Authorization", "Bearer " + token))
            .andExpect(status().isOk());
    }
}
```

## Disabling JWT

To disable JWT authentication (e.g., for local development):

```properties
ftgo.security.jwt.enabled=false
```

When disabled, the `JwtAuthenticationFilter` is not registered, and the security configuration falls back to requiring authentication without JWT validation (useful with `@WithMockUser` in tests or basic auth).
