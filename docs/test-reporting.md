# Test Reporting

This document describes the test report configuration, artifact management, and reporting strategy for the FTGO project's CI test pipelines.

---

## Overview

The FTGO test pipelines generate multiple report formats to support both automated CI decisions and manual debugging. Reports are produced by Gradle's test framework and uploaded as GitHub Actions artifacts.

| Report Type | Format | Purpose | Generated By |
|-------------|--------|---------|--------------|
| JUnit XML | `.xml` | CI system parsing, trend analysis | Gradle (automatic) |
| HTML | `.html` | Human-readable test results | Gradle (automatic) |
| JaCoCo XML | `.xml` | Coverage data for tools (SonarQube, Codecov) | JaCoCo plugin |
| JaCoCo HTML | `.html` | Human-readable coverage reports | JaCoCo plugin |

---

## Report Locations

### Local Build Reports

Gradle generates reports under each module's `build/` directory:

```
<module>/build/
  reports/
    tests/
      test/                    # Unit test HTML report
        index.html
      integrationTest/         # Integration test HTML report (if applicable)
        index.html
    jacoco/
      test.exec                # Binary coverage data
      jacocoTestReport.html    # HTML coverage report
  test-results/
    test/                      # JUnit XML results
      TEST-*.xml
    integrationTest/           # Integration test XML results
      TEST-*.xml
```

### CI Artifact Structure

Artifacts uploaded to GitHub Actions follow a consistent naming convention:

| Artifact Name | Contents | Uploaded When |
|--------------|----------|---------------|
| `<service>-unit-test-reports` | HTML test reports | On test failure |
| `<service>-unit-test-xml` | JUnit XML results | Always |
| `<service>-integration-test-reports` | HTML + XML integration test reports | On test failure |
| `e2e-test-reports` | E2E HTML + XML reports | Always |
| `e2e-application-logs` | Application logs during E2E | On failure |

All artifacts are retained for **14 days**.

## Report Configuration

### Gradle Test Reporting

Test reporting is configured through the Gradle test task and convention plugins. The following settings are applied:

```groovy
// Configured via FtgoTestingConventionsPlugin or individual build.gradle
tasks.withType(Test) {
    // Enable both HTML and JUnit XML reports
    reports {
        html.enabled = true
        junitXml.enabled = true
    }

    // Test logging configuration
    testLogging {
        events 'passed', 'skipped', 'failed'
        showStandardStreams = false
        exceptionFormat = 'full'
    }
}
```

The `IntegrationTestsPlugin` in `buildSrc/` configures integration test reports to use a separate output directory:

```groovy
// From IntegrationTestsPlugin.groovy
tasks.withType(Test) {
    reports.html.destination = project.file("${project.reporting.baseDir}/${name}")
}
```

This ensures unit test and integration test reports do not overwrite each other.

### JUnit XML Format

JUnit XML reports follow the standard schema and include:

- Test class name and method name
- Execution time per test
- Pass/fail/skip status
- Full stack traces for failures
- System output and error streams (if `showStandardStreams = true`)

Example JUnit XML structure:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<testsuite name="net.chrisrichardson.ftgo.orderservice.domain.OrderTest"
           tests="5" skipped="0" failures="1" errors="0" time="0.234">
  <testcase name="shouldCreateOrder" classname="...OrderTest" time="0.045"/>
  <testcase name="shouldCancelOrder" classname="...OrderTest" time="0.012">
    <failure message="Expected CANCELLED but was APPROVED">
      java.lang.AssertionError: ...
    </failure>
  </testcase>
</testsuite>
```

## CI Workflow Report Integration

### GitHub Actions Step Summary

Each test job writes a summary table to `$GITHUB_STEP_SUMMARY`, which appears in the Actions UI:

```markdown
## order-service Unit Tests

| Property | Value |
|----------|-------|
| Status | `success` |
| JDK | `8 (temurin)` |
| Commit | `abc1234` |
```

The summary jobs at the end of each workflow aggregate all service results into a single table.

### Artifact Upload Strategy

Reports are uploaded using `actions/upload-artifact@v4` with the following strategy:

| Condition | What is Uploaded | Why |
|-----------|-----------------|-----|
| Always | JUnit XML results | Enables CI tools to parse results regardless of outcome |
| On failure | HTML test reports | Provides detailed failure information for debugging |
| On failure (E2E) | Application logs | Helps diagnose infrastructure issues |

### Viewing Reports

1. Navigate to the GitHub Actions workflow run
2. Scroll to the **Artifacts** section at the bottom
3. Download the relevant artifact ZIP file
4. Extract and open `index.html` in a browser for HTML reports

## Test Result Aggregation

### Summary Job Pattern

Each workflow includes a summary job that aggregates results:

```yaml
unit-test-summary:
  needs: [detect-changes, test-order-service, test-consumer-service, ...]
  if: always()
  steps:
    - name: Check test results
      run: |
        # Iterates over all job results
        # Fails if any triggered job failed
        # Writes aggregate summary table
```

This provides:
- A single pass/fail status for the entire test tier
- A summary table showing which modules were tested and their results
- Clear error messages when tests fail

### Cross-Workflow Visibility

The three test workflows (`test-unit.yml`, `test-integration.yml`, `test-e2e.yml`) run independently. To get a complete view:

1. Check the **Actions** tab for the branch/PR
2. All three workflows will appear as separate checks
3. Each provides its own summary in the PR status checks

## Future Enhancements

### Test Result Dashboard

When the project adopts a test analytics tool, JUnit XML artifacts can be consumed by:

- **SonarQube** — Import JUnit XML for test tracking
- **Codecov / Coveralls** — Import JaCoCo XML for coverage trends
- **Allure Framework** — Generate rich interactive test reports
- **Datadog CI Visibility** — Track test performance over time

### PR Comment Reports

Future CI enhancements can add test result summaries as PR comments using actions like:

- `dorny/test-reporter` — Parses JUnit XML and posts results as PR checks
- `EnricoMi/publish-unit-test-result-action` — Summarizes test results in PR comments

### Flaky Test Detection

Track test stability over time by:

1. Collecting JUnit XML results from every run
2. Identifying tests that intermittently fail
3. Quarantining flaky tests until fixed
